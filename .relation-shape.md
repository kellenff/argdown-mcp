## In-memory shape (from `@argdown/core/dist/model/model.d.ts`)

```ts
export interface IRelation extends HasColor {
  type: ArgdownTypes.RELATION; // literal "relation"
  from?: RelationMember; // IArgument | IEquivalenceClass | IInference
  to?: RelationMember;
  relationType: RelationType; // "attack" | "support" | "entails" | "contrary" | "contradictory" | "undercut"
  occurrences: IRuleNode[];
}

export type RelationMember = IArgument | IEquivalenceClass | IInference;
```

Each `RelationMember` carries `.title: string` (via `HasTitle`) and `.type: ArgdownTypes` (via the per-interface discriminant):

- `IArgument.type === "argument"`
- `IEquivalenceClass.type === "equivalence-class"`
- `IInference.type === "inference"`

`RelationType.ATTACK = "attack"` is the canonical discriminant for the relation kind.

## JSON-export serialisation

`JSONExportPlugin` flattens `RelationMember` to `{ <name>: title, <name>Type: kind }`:

```json
{
  "type": "relation",
  "relationType": "attack",
  "from": "B",
  "fromType": "argument",
  "to": "A",
  "toType": "argument"
}
```

The dung algorithm consumes the in-memory shape (via `runInline`/`runFile`), not the JSON form. The JSON form is for diagnostic / probe use only.

### `three_cycle_pcs` — `<A>\n  - <B>` ... `<B>\n  - <C>` ... `<C>\n  - <A>`

- arguments: `[A, B, C]`
- relations: 3 attacks: `B→A`, `C→B`, `A→C`
- ✅ 3-cycle works under the inline `-` syntax

### `statement_attacks_only` — `[s1]\n  - [s2]`

- arguments: `[]`
- relations: 1 attack with `fromType: "equivalence-class"`, `toType: "equivalence-class"`
- ✅ statement-level attacks have `fromType !== "argument"` → cleanly filtered by argument-discriminant

### `mixed_both` — combination

- arguments: `[A, B]`
- relations:
  - `support` `[p] → A` (equivalence-class → argument)
  - `attack` `B → A` (argument → argument)
  - `attack` `[s2] → [s1]` (equivalence-class → equivalence-class)
- ✅ **Statement attacks are NOT auto-lifted into argument attacks.** The mixed-mode document keeps statement-level and argument-level relations distinct in the response array.

## Discriminant for `extractArgumentAttacks`

```ts
function extractArgumentAttacks(
  resp: IArgdownResponse,
): Array<[string, string]> {
  const out: Array<[string, string]> = [];
  for (const r of resp.relations ?? []) {
    if (r.relationType !== RelationType.ATTACK) continue;
    if (r.from?.type !== ArgdownTypes.ARGUMENT) continue;
    if (r.to?.type !== ArgdownTypes.ARGUMENT) continue;
    const fromTitle = (r.from as IArgument).title;
    const toTitle = (r.to as IArgument).title;
    if (fromTitle && toTitle) out.push([fromTitle, toTitle]);
  }
  return out;
}
```

Edge direction: `[from, to]` = `[attacker, attacked]`. This matches Dung convention (`a` attacks `b` ⇒ `(a, b) ∈ R`).

Undercut (`relationType === "undercut"`) targets `IInference`, never `IArgument`, so it is filtered out by the type-guard. v0.3 deliberately ignores undercuts — they belong in a future structured-argumentation layer.

Contrary / contradictory (`relationType === "contrary" | "contradictory"`) operate at the equivalence-class level only and are filtered out. v0.3 honours only `RelationType.ATTACK` between arguments.

## Plan adjustments required (downstream of Task 0)

1. **Fixture syntax correction:** The plan's SC-2 example `<A>: claim. <B>: claim. <A> > <B>` uses top-level `>` which is NOT valid Argdown attack syntax. The correct syntax is the inline `-` form under the target argument's body:

   ```
   <A>: claim.
     - <B>: claim.
   ```

   meaning "B attacks A". All Dung fixtures (T4) must use this form.

2. **`extractArgumentAttacks` location:** Lives in `src/argdown/dung.ts` (per plan), consumes the in-memory `IArgdownResponse`. The `IArgument` cast is safe because the type-guard checks `r.from?.type === "argument"`.

3. **No JSON-export coupling:** The Dung algorithm path does NOT depend on `JSONExportPlugin`; `runInline(source, { withJson: false })` is sufficient. This keeps `dung_extensions` independent of the JSON export plugin's serialisation.
