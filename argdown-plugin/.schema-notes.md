## `.claude-plugin/plugin.json` — required fields

| Field  | Type   | Notes                                                                                                                                                                           |
| ------ | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `name` | string | Kebab-case, unique across installed plugins. No spaces or special characters. Example: `"argdown-plugin"`. This is the only field present in every single observed plugin.json. |

Evidence: Every exemplar — including minimal Anthropic plugins like `plugin-dev`, `skill-creator`, `mcp-server-dev`, `code-review`, `commit-commands` — contains only `name` (plus optional fields). The plugin-structure SKILL.md confirms: `{ "name": "plugin-name" }` is the minimal manifest.

| Field              | Type               | Notes                                                                                                                                                                                                 |

## Skills discovery contract

**Auto-scanned vs. manifest-declared?** AUTO-SCANNED.

Evidence from plugin-structure SKILL.md (authoritative Anthropic-authored reference):

> "Auto-Discovery: All `SKILL.md` files in skill subdirectories load automatically"
>
> "Claude Code automatically discovers skills: Scans `skills/` directory, Finds subdirectories containing `SKILL.md`, Loads skill metadata (name + description) always, Loads SKILL.md body when skill triggers, Loads references/examples when needed"

Evidence from skill-development SKILL.md:

> "Claude Code automatically discovers skills: Scans `skills/` directory, Finds subdirectories containing `SKILL.md`"

No observed plugin.json contains a `skills` array or any skills-related field — confirmed by examining all 10+ exemplars. Skills are discovered purely by filesystem convention.

### Auto-scanned layout

The expected structure is `skills/<name>/SKILL.md` (subdirectory, not flat):

```
skills/
└── skill-name/          ← subdirectory
    └── SKILL.md         ← required filename (exactly "SKILL.md", not "README.md")
```

Flat files (`skills/skill-name.md`) are NOT the pattern. Each skill must be its own subdirectory containing `SKILL.md`.

### Optional skill subdirectories (also auto-loaded as needed)

```
skills/skill-name/
├── SKILL.md             ← required
├── references/          ← optional, loaded on demand
├── examples/            ← optional
└── scripts/             ← optional
```

### Opt-out / naming conventions

No opt-out mechanism observed. Any subdirectory inside `skills/` that contains a `SKILL.md` will be registered. The SKILL.md frontmatter's `description` field controls _when_ the skill triggers (not whether it loads). There is no observed file-naming exclusion convention.

### SKILL.md frontmatter (required fields)

| Field         | Type   | Notes                                                                              |
| ------------- | ------ | ---------------------------------------------------------------------------------- |
| `name`        | string | Skill display name                                                                 |
| `description` | string | Trigger phrases in third-person format; determines when Claude activates the skill |

Optional frontmatter: `version` (string, semver).

## MCP server registration (.mcp.json)

### Option A: Separate `.mcp.json` at plugin root

Most plugins use a standalone `.mcp.json` file. Two observed wrapper shapes:

**With `mcpServers` wrapper** (tool-executor, semgrep):

```json
{
  "mcpServers": {
    "server-name": {
      "type": "stdio",
      "command": "node",
      "args": ["${CLAUDE_PLUGIN_ROOT}/dist/index.js"]
    }
  }
}
```

**Without `mcpServers` wrapper** — bare server map (context7, serena, playwright):

```json
{
  "context7": {
    "command": "npx",
    "args": ["-y", "@upstash/context7-mcp"]
  }
}
```

Both patterns appear in official plugins. The `mcpServers` wrapper is consistent with the standard `.claude/mcp.json` format; the bare map is also accepted.

### Option B: `mcpServers` inline in plugin.json

`chrome-devtools-mcp` embeds the server directly in plugin.json:

```json
{
  "name": "chrome-devtools-mcp",
  "version": "0.22.0",
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest"]
    }
  }
}
```

### MCP server entry fields

| Field     | Type     | Required    | Notes                                                                                                         |
| --------- | -------- | ----------- | ------------------------------------------------------------------------------------------------------------- |
| `command` | string   | Yes (stdio) | Executable: `node`, `npx`, `uvx`, `semgrep`, etc.                                                             |
| `args`    | string[] | No          | Arguments array. Use `${CLAUDE_PLUGIN_ROOT}/...` for local entry points.                                      |
| `type`    | string   | No          | `"stdio"` (explicit in tool-executor). Default when absent is also stdio. `"http"` observed in github plugin. |
| `url`     | string   | Yes (http)  | Used when `type: "http"` — e.g. `"https://api.githubcopilot.com/mcp/"`.                                       |
| `headers` | object   | No          | HTTP headers map; used with `type: "http"` for auth tokens.                                                   |
| `env`     | object   | No          | Environment variable map; documented in plugin-structure SKILL.md.                                            |

### Local Node entry-point pattern (relevant to argdown-plugin)

The tool-executor exemplar shows the canonical local-path approach for a Node/TS server:

```json
{
  "mcpServers": {
    "tool-executor": {
      "type": "stdio",
      "command": "node",
      "args": ["${CLAUDE_PLUGIN_ROOT}/dist/index.js"]
    }
  }
}
```

`${CLAUDE_PLUGIN_ROOT}` is resolved at plugin load time to the plugin's installed directory. This means the build artifact (`dist/index.js`) must be committed/shipped with the plugin — it will not be built at install time.

## Plugin package.json

No Claude Code-specific fields were observed in any plugin's `package.json`. The tool-executor's `package.json` is a standard npm package with no `claudePlugin`, `plugin`, or other plugin-specific keys. The superpowers `package.json` is minimal (`name`, `version`, `type`, `main`) and also has no plugin-specific fields.

The chrome-devtools-mcp `package.json` has `"mcpName": "io.github.ChromeDevTools/chrome-devtools-mcp"` — this appears to be an MCP registry identifier, not a Claude Code plugin field.

**Conclusion**: Plugin packaging metadata lives entirely in `.claude-plugin/plugin.json`. No `package.json` fields are required or used by the Claude Code plugin loader.

## Open questions / gaps

1. **Bare vs. wrapped `.mcp.json`**: Two schemas are observed (`mcpServers` wrapper vs. bare server map). It is unclear whether one is preferred or whether both are equally supported. The wrapped form matches the Claude Code settings schema; the bare form may be a legacy or simplified variant.

2. **`commands`, `agents`, `mcpServers` custom path resolution**: The plugin-structure SKILL.md states custom paths "supplement" rather than replace defaults. The exact merge/dedup behavior is undocumented in the exemplars.

3. **Version field in `"unknown"` cache paths**: Many Anthropic official plugins have `"unknown"` as their cache path version and omit the `version` field in plugin.json. It is unclear if this is intentional (Anthropic omits versions) or an artifact of their publishing pipeline.

4. **No `mcpServers` reference path field observed**: The plugin-structure SKILL.md documents `"mcpServers": "./.mcp.json"` as a string path in plugin.json, but no exemplar actually uses this form — all use either inline `mcpServers` object or a separate `.mcp.json` file. Whether the string-path form is supported needs verification.

5. **Dist artifact shipping**: The local-Node MCP pattern requires `dist/` to be committed. Whether plugins are expected to ship pre-built or whether an install hook exists is not addressed by any exemplar.
